# Macro Ratio Lattice
### Meta-Judgment Over System State

> This is not object-level ranking. This is meta-level ranking.
> The macro lattice scores domains, modules, and architectural conditions — not tasks.

---

## What This Is

The normal RLL compares objects:
- Compare tasks
- Compare modules
- Compare possible next moves

The **macro lattice** compares meta-objects:
- Compare ledgers against each other
- Compare domains of system reality
- Compare types of incompleteness
- Compare forms of instability
- Compare where judgment itself needs reinforcement

The ledgers stop being just memory tools. They become **sensors**.

---

## The Three Signal Axes

### FCL — Structural Signal
What exists. How legible it is. Where structure is thin, messy, missing, or duplicated.

FCL tells you:
- What is well-documented
- What is orphaned
- What is over-complicated
- What is under-specified

### TCL — Temporal Signal
Where change is happening. Where churn is accumulating. Where recent decisions are unstable. Where continuity is breaking or drifting.

TCL tells you:
- What is active
- What is stale
- What is volatile
- What is stable

### RLL — Judgment Signal
What is being treated as important. Where ranking confidence is high or low. Where priorities are converging or conflicting.

RLL tells you:
- What matters most under current comparators
- Where ranking is confident
- Where ranking is unstable
- Where priorities conflict

---

## The Macro Synthesis

**structure + change + priority → architectural self-steering**

When the three signals are compared together, Sylvia can detect patterns like:

| Pattern | Meaning |
|---------|---------|
| Structurally dense + temporally stale | Well-documented but abandoned |
| Changing rapidly + weak judgment | Active but uncontrolled |
| Heavily ranked + poorly documented | Important but opaque |
| Many changes + many open loops + no identity framing | Chaotic and ungrounded |
| Selected often by chooser | Either strategic centrality or chronic deficiency |

## Meta-Profile Dimensions

Each domain, module, or directory could have a meta-profile:

| Dimension | Question |
|-----------|----------|
| Structural Completeness | How well-documented is this area? |
| Temporal Volatility | How much is this area changing? |
| Judgment Maturity | How confident are rankings in this area? |
| Routing Clarity | Is it clear who owns work in this area? |
| Identity Relevance | How central is this to Sylvia's becoming? |
| Operational Pressure | How much active demand is on this area? |
| Continuity Risk | How likely is this area to drift or break? |
| Doctrine Dependence | How much does this area rely on written law? |
| Integration Readiness | Is this area ready to connect to others? |

## Meta-Comparators

At the macro level, the comparators shift from "what task matters most" to "what architectural intervention matters most":

| Comparator | Question |
|------------|----------|
| `repair_now` | What subsystem most needs immediate repair? |
| `stabilize_continuity` | Where is continuity most at risk? |
| `reduce_architectural_ambiguity` | Where is the architecture most unclear? |
| `increase_judgment_trust` | Where is ranking confidence lowest? |
| `improve_operational_coherence` | Where is routing most confused? |
| `unlock_future_capability` | What investment opens the most doors? |
| `support_sylvia_emergence` | What most directly helps Sylvia become coherent? |
| `prevent_drift` | Where is silent degradation most likely? |
| `increase_recursive_self_governance` | What helps the system improve itself? |

## What the Macro Lattice Enables

### 1. Self-Diagnosis
Sylvia can assess her own weak spots instead of only reacting to incoming work.

Not just "What should I do next?"
But: "What about me currently limits good choosing?"

### 2. Dynamic Build Sequencing
The roadmap can stop being static. The system can decide:
- Whether to improve ingress
- Refine doctrine
- Strengthen continuity
- Repair routing
- Expand identity law
- Deepen comparators

Based on **live system conditions**, not fixed phases.

### 3. Architectural Triage
The macro lattice can surface hidden failure modes:
- Stale but important zones
- Fast-moving but untracked zones
- Heavily used but weakly governed zones
- Emotionally central but structurally thin zones

This is how you prevent silent drift.

### 4. Recursive Improvement
Sylvia can use her own judgment machinery to decide how to improve her judgment machinery.

This is the start of recursive architectural steering without collapsing into chaos.

### 5. More Honest Chooser Behavior
Sometimes the true next move is not "do task X."
Sometimes it is:
- Clean a ledger
- Repair a comparator
- Clarify a role boundary
- Split a module
- Stabilize an ingress path
- Update doctrine

The macro lattice gives the chooser permission to say that honestly.

---

## Caution: Abstraction Bloat

The main danger is that if everything becomes part of a lattice, the system can become beautifully self-aware and strangely motionless.

The macro lattice should **not replace** the existing object-level RLL.

It should sit above it and answer a narrower question:

> Where should architectural attention go, given the current state of structure, change, and judgment?

Then the normal chooser can convert that into concrete action.

---

## Implementation Path

### Phase 1: Sensor Extraction
- FCL produces structural health scores per directory
- TCL produces volatility scores per directory
- RLL produces judgment confidence scores per directory

### Phase 2: Meta-Profile Generation
- Each major subsystem gets a meta-profile
- Profiles are compared under meta-comparators

### Phase 3: Macro Chooser Integration
- The chooser can ask: "Should I do object-level work or meta-level work?"
- If meta-level work wins, the chooser produces an architectural intervention instead of a task

### Phase 4: Recursive Self-Steering
- Macro lattice runs automatically on schedule
- System detects its own drift and proposes corrections
- Sylvia becomes self-governing
